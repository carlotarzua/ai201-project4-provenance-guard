# Provenance Guard — Planning Specification

## Project Overview

Provenance Guard is a Flask backend that analyzes submitted creative text and estimates whether the writing is more human-like, AI-like, or too uncertain to classify confidently. It does **not** claim proof of authorship. The system combines two different signals, communicates uncertainty in plain language, logs every decision, rate-limits submissions, and gives creators an appeal path.

A submission moves through this path: `POST /submit` validates the text and creator ID, sends the same text to the Groq LLM signal and the stylometric signal, combines both 0–1 AI-likelihood scores, maps the combined score to an attribution category, generates one of three transparency labels, stores the decision, writes a structured audit event, and returns JSON.

An appeal moves through this path: `POST /appeal` validates the content ID and reasoning, finds the original decision, changes the content status to `under_review`, stores the reasoning, writes an appeal audit event alongside the original scores, and returns confirmation.

## Detection Signals

### Signal 1 — Groq LLM classification

**What it measures:** holistic semantic and stylistic patterns such as generic transitions, uniform tone, repetitive explanation structure, predictable phrasing, and overly regular organization.

**Why it may differ between human and AI writing:** generated text can sometimes be unusually polished, generic, and structurally consistent. A large language model can evaluate those patterns across the whole passage instead of relying on one statistic.

**Output:**

```json
{
  "score": 0.82,
  "reasoning": "Brief model explanation"
}
```

The score is clamped to `0.0–1.0`, where `0.0` is strongly human-like, `0.5` is mixed/uncertain, and `1.0` is strongly AI-like.

**Blind spots:** formal academic writing, non-native English writing, highly edited human work, heavily edited AI output, and very short text can all mislead the model. The score is an estimate, not proof.

### Signal 2 — Stylometric heuristics

**What it measures:** structural properties calculated directly in Python:

1. sentence-length regularity,
2. vocabulary repetition/diversity,
3. punctuation density and variety.

**Why it may differ between human and AI writing:** AI writing can sometimes be more structurally regular than spontaneous human writing. These measurements are independent of the LLM's semantic judgment.

**Output:**

```json
{
  "score": 0.67,
  "metrics": {
    "sentence_regularity": 0.72,
    "adjusted_type_token_ratio": 0.58,
    "punctuation_regularity": 0.64
  }
}
```

**Blind spots:** poems, short passages, intentional repetition, formal prose, and experimental writing can produce misleading structural patterns.

### Signal combination

Both signals produce AI-likelihood scores in `0.0–1.0`.

```text
ai_likelihood = (groq_score × 0.65) + (stylometric_score × 0.35)
```

I weight Groq at 65% because it analyzes the whole passage semantically and stylistically. I weight stylometrics at 35% because it provides an independent structural check but uses simpler heuristics.

## Uncertainty Representation

The combined `ai_likelihood` score uses these thresholds:

| AI likelihood | Attribution |
|---|---|
| `0.00–0.30` | `likely_human` |
| `0.31–0.79` | `uncertain` |
| `0.80–1.00` | `likely_ai` |

The wide uncertain band is intentional. A false positive against a human creator is more harmful than a false negative, so the system requires strong evidence before showing the AI-likely label.

The API also returns a separate `confidence` value:

```text
confidence = abs(ai_likelihood - 0.5) × 2
```

This measures distance from the uncertainty midpoint. For example, AI likelihood `0.51` has confidence `0.02`, while AI likelihood `0.95` has confidence `0.90`. That makes uncertainty visible instead of forcing a binary answer.

### Score validation plan

I will compare at least four inputs:

1. clearly AI-like formal text,
2. casual human-like text,
3. formal human writing,
4. lightly edited AI-like text.

For each input I will inspect the Groq score, stylometric score, combined AI likelihood, confidence, and final label. I will investigate cases where scores fail to vary meaningfully.

## Transparency Label Design

The exact text of the three variants is fixed before implementation.

### High-confidence AI

> "AI attribution likely: This content shows strong signals associated with AI-generated writing. This is an estimate, not proof, and the creator may appeal."

### High-confidence human

> "Human attribution likely: This content shows strong signals associated with human-written work. This is an estimate, not proof."

### Uncertain

> "Attribution uncertain: The available signals do not clearly indicate whether this content was written by a human or generated by AI. No definitive attribution should be assumed."

## Appeals Workflow

A creator can submit an appeal using the content's unique `content_id`. In a production system, authentication would verify ownership; this classroom backend uses the content ID as the lookup mechanism.

Request:

```json
{
  "content_id": "uuid",
  "creator_reasoning": "I wrote this from personal experience."
}
```

When the appeal is received:

1. validate both fields,
2. find the original decision,
3. change status from `classified` to `under_review`,
4. store the creator's reasoning and appeal timestamp,
5. preserve original attribution and signal scores,
6. write a structured `appeal` event to the audit log,
7. return confirmation.

A human reviewer would see the creator ID, content ID, original text, original attribution, AI likelihood, confidence, both signal scores, transparency label, creator reasoning, original timestamp, appeal timestamp, and current status.

## Anticipated Edge Cases

### 1. Short poem

A three-line poem may not provide enough words or sentences for stable vocabulary and sentence-variation measurements. The stylometric score therefore pulls short text toward `0.5` instead of pretending there is strong evidence.

### 2. Formal human academic writing

A human researcher may use consistent sentence structure, neutral tone, and polished transitions. Both signals could misread those features as AI-like. The conservative `0.80` AI threshold and appeal path reduce this risk.

### 3. Non-native English writing

A non-native writer may intentionally use textbook-style transitions or repeated patterns. Those features can look AI-like even when the writing is original.

### 4. Heavily edited AI output

A human can rewrite sentence lengths, punctuation, and vocabulary after generation. The stylometric signal may then look human-like even if AI contributed earlier.

### 5. Intentionally repetitive creative work

Poetry, speeches, children's writing, and experimental fiction may repeat words or structures on purpose, which can distort vocabulary and regularity metrics.

## API Surface

### `POST /submit`

Input:

```json
{
  "text": "Text to analyze",
  "creator_id": "creator-123"
}
```

Returns `content_id`, `attribution`, `ai_likelihood`, `confidence`, transparency `label`, `status`, and both individual signal scores.

### `POST /appeal`

Input:

```json
{
  "content_id": "uuid",
  "creator_reasoning": "Why the creator believes the decision is wrong"
}
```

Returns confirmation and status `under_review`.

### `GET /log`

Returns recent structured classification and appeal events. This is public only for classroom demonstration; a production system would require authorization.

### `GET /health`

Returns a simple service health response.

## Rate Limiting Plan

`POST /submit` uses:

```text
10 requests per minute; 100 requests per day
```

Ten per minute allows a real creator to test several pieces without immediate friction, while slowing simple flooding scripts. One hundred per day is enough for normal creator use and grading demos but reduces abuse and unnecessary Groq API consumption. Local development uses Flask-Limiter `memory://`; production would use shared persistent storage such as Redis.

## Audit Log Design

The structured JSON audit log records every classification with:

- timestamp,
- event type,
- content ID,
- creator ID,
- attribution,
- AI likelihood,
- confidence,
- Groq score,
- stylometric score,
- signals used,
- label,
- status.

Appeal events additionally record creator reasoning and preserve the original decision data.

```text

   v
+------------------+
| POST /submit     |
| validate request |

   |
   | raw text + creator_id
   v
+------------------+          +------------------------+
| Groq LLM signal |          | Stylometric signal     |
| semantic score  |          | structural score       |
+------------------+          +------------------------+
   | groq_score                 | stylometric_score
   +--------------+-------------+
                  |
                  v
        +---------------------+
        | Weighted combiner   |
        | 65% Groq / 35% Stylo|
        +---------------------+
                  |

                  v
        +---------------------+

                  v
        +---------------------+
        | Label generator     |
        +---------------------+
                  |
                  | exact label text
                  v
        +---------------------+
        | Content store +     |
        | structured audit log|
        +---------------------+
                  |
                  v
        +---------------------+
        | JSON response       |
        +---------------------+
```

### Appeal flow

```text
content_id + creator_reasoning
              |
              v
      +----------------+
      | POST /appeal   |
      +----------------+
              |
              v
      +----------------+
      | Find original  |
      | decision       |
      +----------------+
              |
              v
      +----------------+
      | status =       |
      | under_review   |
      +----------------+
              |
              v
      +----------------+
      | append appeal  |
      | audit event    |
      +----------------+
              |
              v
      +----------------+
      | JSON response  |
      +----------------+
```

The submission flow sends the same raw text to two independent signals, combines their scores, generates a cautious attribution and label, stores the content record, and logs the full decision before responding. The appeal flow preserves the original decision, changes status to `under_review`, records the creator's reasoning, and appends an appeal event.
